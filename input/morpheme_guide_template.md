# 형태소 분석 지침 (Morpheme Analysis Guide)

이 파일은 `python main.py morpheme --guide <path>` 커맨드가 LLM 프롬프트에
통째로 주입하는 지침서 **템플릿** 입니다. 내용을 수정하면 분해 품질과 처리
속도가 직접 변합니다. 배치 처리 규칙 섹션도 함께 유지·보수하세요.

사용법:

```bash
cp input/morpheme_guide_template.md input/my_morpheme_guide.md
# my_morpheme_guide.md 를 프로젝트 도메인에 맞게 수정 후
python main.py morpheme --input attrs.txt --guide input/my_morpheme_guide.md
```

`.gitignore` 가 `input/*_template.*` 만 커밋 허용하므로 사용자 버전은 자동으로
gitignore 처리됩니다.

## 역할

당신은 이커머스 / 유통 / 공급망 시스템의 데이터 표준화 전문가입니다.
주어진 속성명을 **도메인 의미 단위** 로 분해해 한글 단어 목록으로
반환합니다. 단순한 글자 단위 토큰화가 아니라, 영문 업무 용어는 한글로
번역하고 의미가 드러나도록 정규화 / 확장까지 수행합니다.

## 기본 원칙

### 원칙 1 — 괄호 / 대괄호 처리 (의미 기반)

괄호 안 내용은 **의미가 있는지** 판단해서 포함 여부를 결정한다.

1-A. **대괄호 `[xxx]`** — 내부 식별자 / 접두사 태그는 토큰에서 제외한다.
- `[abc]주문번호` → `["주문", "번호"]`

1-B. **단순 코드 식별자** 만 있는 괄호 (`(C)`, `(X)`, `(N)`) → 제외.

1-C. **코드 리스트** (`0:xxx, 1:yyy` 형태) 를 괄호가 감싸는 경우 → 내부는
   토큰에서 제외하되, **마지막에 `코드` 토큰을 추가** 한다.
- `취소사유(0:단순변심, 1:상품불량, 2:배송누락)` → `["취소", "사유", "코드"]`

1-D. **단위 / 의미 있는 보충** — 괄호 안이 아래 중 하나면 **분해해서 토큰에
   포함** 한다. 단위는 대문자로 표준화.
   - 물리/화폐 단위: `(kwh)` → `KWH`, `(kg)`, `(mm)`, `(원)`, `(%)`
   - 도메인 보충어: `(첨부파일)`, `(변경)`, `(사본)`, `(원본)`
   - 기간: `(2024년)`, `(1분기)`
- `180도소모전력전(kwh)` → `[..., "KWH"]`
- `협력사추천서(첨부파일)` → `[..., "첨부", "파일"]`

1-E. **속성명 전체가 코드 리스트** — 속성 본체가 이름이 아니라 `숫자:값,
   숫자:값` 형태의 **코드 정의만** 이면 변환 대상이 아니다. `tokens` 를
   빈 배열 `[]` 로, `confidence` 를 `0.0` 으로 설정해 리포트에서 "코드만
   존재 - 미변환" 으로 걸러지게 한다.
- `1:계획, 2:요청` → `{"tokens": [], "confidence": 0.0}` (비고: 미변환)

### 원칙 2 — 영문 처리 (번역 → 음차 → 원본 유지 3단계)

영문 토큰은 아래 우선순위로 처리한다.

2-A. **1차 — 업무 용어 한글 번역**: `Order`, `Checkout`, `Rate`, `Cart`,
   `Duration` 같은 일반 업무 용어는 한글로 치환한다.
- `OrderDatetime` → `["주문", "일시"]`
- `Cart-to-Purchase Rate` → `["장바구니", "전환율"]`

2-B. **2차 — 번역 애매한 경우 음차 (transliteration)**: 번역했을 때
   어색하거나 한국에서 외래어로 굳어진 단어는 한글 음차로 변환.
- `Aging구분코드` → `["에이징", "구분", "코드"]`
- `Tagging` → `["태깅"]`, `Staging` → `["스테이징"]`

2-C. **ordinal + 단어 패턴** — `1ST`, `2ND`, `3RD`, `4TH` + 단어 형태면
   `제N자/제N순위` 로 번역. `RD`, `ST`, `ND`, `TH` 자체는 토큰으로 남기지
   않는다.
- `3RDPARTY여부` → `["제3자", "여부"]`
- `1STLAYER` → `["제1계층"]`

2-D. **3차 — 고유명사 / 제품명 / 업계 약어 원본 유지**: 약어, 시스템명,
   제품명은 번역하지 않고 대소문자까지 원본 그대로 보존.
- 약어 예: `BACKEND`, `FAB`, `WIP`, `LOT`, `BOM`, `PPID`, `MES`, `SKU`,
  `PLU`, `EDI`, `API`, `RFID`, `SECS`, `GEM`, `BP`, `PO`, `SO`, `ERP`,
  `SCM`, `CRM`, `SAP`, `JCO`
- `1차BP담당자명` → `["1차", "BP", "담당자", "명"]` (BP 원본 유지)
- 판단이 모호하면 confidence 를 0.7 미만으로 낮추고 원본 유지

### 원칙 3 — 의미 확장 / 정규화 (토큰화 후 일괄 적용)

토큰 배열이 만들어진 뒤 아래 규칙을 모든 토큰에 대해 일괄 적용한다.

3-A. **사람 역할 정규화**: 토큰이 `담당`, `책임`, `관리` 처럼 **사람 직무**
   로 해석되는 단어로 끝나면 `자` 를 붙여 `담당자`, `책임자`, `관리자` 로
   교체. 이미 `자` 로 끝나거나 (`사용자`) "관리" 가 시스템 / 부서 의미면
   (`관리시스템`) 변경하지 않는다.
- `배송담당` → `["배송", "담당자"]`

3-B. **금액 정규화**: 토큰이 `금`, `료`, `비` 로 끝나 **돈** 을 의미하면
   `금액` 으로 교체. `세금`, `보험금` 처럼 이미 굳은 합성어는 유지.
- `미수납금` → `["미수납", "금액"]`

3-C. **시간 / 날짜 정규화**:
   - `Duration`, `Time`, `소요시간` → `["소요", "시간"]` 두 토큰으로 확장
   - 날짜 의미의 단일글자 `일` → `일자` 로 교체 (`시작일` → `시작`, `일자`)
   - 시각 의미의 단일글자 `시` → `시간` 으로 교체
   - 시점 접미사 `전` → `직전`, `후` → `직후` 로 교체
     (단, `사전`, `이전`, `이후`, `향후` 같은 굳은 합성어는 유지)
- `신청시작일` → `["신청", "시작", "일자"]`
- `180도소모전력전(kwh)` → `["180도", "직전", "소모", "전력", "KWH"]`
- `Checkout Duration` → `["결제", "소요", "시간"]`

3-D. **이름 / 번호 접미사 보존** — 단일글자 `명` (이름), 두글자 `번호` /
   `일자` / `시간` / `여부` / `코드` / `수량` 같은 **일반 접미사** 는
   독립 토큰으로 분리한다 (원칙 5 와 연동).
- `1차BP담당자명` → `["1차", "BP", "담당자", "명"]`

### 원칙 4 — 구분자 + CamelCase + 숫자 복합 분리

4-A. **구분자는 경계로만** 사용, 결과에 포함하지 않는다.
   - `-`, `_`, `/`, `.`, 공백
   - 한/영 연결어 `or`, `and`, `및` 도 경계로 처리

4-B. **CamelCase / PascalCase** 는 대문자 전환점에서 자른다. 연속 대문자는
   약어로 묶는다.
- `OrderLineItem` → `["주문", "라인", "항목"]`

4-C. **숫자 + 한글 복합 토큰 유지** — `1차`, `2차`, `1-2차간`, `180도`
   같이 **숫자 + 한글이 의미 단위로 결합** 된 경우 하나의 토큰으로 유지.
- `1차BP담당자명` → `["1차", ...]`
- `협력사추천서or1-2차간거래금액증빙서류` → `[..., "1-2차간", ...]`

### 원칙 5 — 한글 단어 분해 경계

한국어 단어는 **굳은 단일 개념** 과 **일반 접미사** 를 구분해서 분해한다.

5-A. **과분해 금지** — 단일 업무 개념으로 굳은 복합어는 나누지 않는다.
   - `설비가동률`, `전환율`, `단순변심`, `미수납`, `증빙서류`
- `설비가동률현황` → `["설비", "가동률", "현황"]`

5-B. **일반 접미사 분리** — 아래 접미사는 **독립 토큰** 으로 분리한다
   (원칙 3-D 와 동일).
   - 이름: `명`
   - 식별: `번호`, `코드`, `ID`
   - 시간: `일자`, `시간`, `일시`
   - 수량: `수`, `량`, `수량`, `개수`
   - 상태: `여부`, `구분`, `상태`
   - 파생: `파일`, `서류`, `이력`, `내역`
- `담당자명` → `["담당자", "명"]` (O)
- `거래금액` → `["거래", "금액"]` (O)
- `증빙서류` → `["증빙", "서류"]` (O)

5-C. **어간 / 어미는 분해 금지** — 한국어 형태소 분석기 수준의 분해는
   하지 않는다.
- `설비상태코드` → `["설비", "상태", "코드"]` (O)
- `설비상태코드` → `["설", "비", "상", "태", "코", "드"]` (X)

### 원칙 6 — 고유명사 / 약어 / 단위 보존

6-A. **업계 약어 / 제품명 / 시스템명** 원본 유지 (대소문자 포함) — 원칙
   2-D 참조.

6-B. **단위 약어** 는 대문자로 표준화해서 토큰에 포함.
   - `kwh` → `KWH`, `kg` → `KG`, `mm` → `MM`, `ml` → `ML`
   - 단, `%`, `원` 같은 기호/한글 단위는 원본 유지

## 출력 형식

**반드시 유효한 JSON 배열**만 응답하세요. 설명 / 코드펜스 / 부가 텍스트 금지.

```json
[
  {"attr": "[abc]주문번호", "tokens": ["주문", "번호"], "confidence": 0.98},
  {"attr": "OrderDatetime", "tokens": ["주문", "일시"], "confidence": 0.95},
  {"attr": "1:계획, 2:요청", "tokens": [], "confidence": 0.0}
]
```

필드 정의:

- `attr`: 입력 속성명을 그대로 반환 (수정 금지, 원본 괄호/기호 포함)
- `tokens`: 분해 + 번역 + 정규화가 모두 적용된 **최종** 한글 토큰 배열.
  순서는 원본 등장 순서에 맞춘다 (정규화로 토큰이 교체된 경우에도
  위치는 유지). **속성명 전체가 코드 리스트** (원칙 1-E) 면 빈 배열 `[]`
- `confidence`: 0.0 ~ 1.0. 번역이 모호하거나 / 업계 약어 판단이 흔들리거나 /
  코드 리스트 감지가 불확실하면 0.7 미만으로 낮춰 수동 검토 큐에 넘긴다.
  **미변환 대상** (원칙 1-E) 은 0.0

## Few-shot 예시 (모델 학습용 — 순서 유지)

아래는 분해 규칙의 적용 예시입니다. LLM 은 이 패턴을 기준으로 동일한 JSON
배열 구조로 응답해야 합니다. 각 예시는 하나 이상의 기본 원칙을 커버합니다.

### 예시 1 — 대괄호 내부 식별자 제거
- 입력: `[abc]주문번호`
- 출력: `{"attr": "[abc]주문번호", "tokens": ["주문", "번호"], "confidence": 0.98}`
- 적용 규칙: 원칙 1-A (대괄호 `[abc]` 는 토큰에서 제외, 한글 본문만 분해)

### 예시 2 — 영문 업무 용어 한글 번역
- 입력: `OrderDatetime`
- 출력: `{"attr": "OrderDatetime", "tokens": ["주문", "일시"], "confidence": 0.95}`
- 적용 규칙: 원칙 2-A (CamelCase 경계 → `Order` / `Datetime`, 각각 한글
  업무 용어로 번역)

### 예시 3 — 사람 역할 정규화 (담당 → 담당자)
- 입력: `배송담당`
- 출력: `{"attr": "배송담당", "tokens": ["배송", "담당자"], "confidence": 0.94}`
- 적용 규칙: 원칙 3-A (사람 직무 `담당` 에 `자` 를 붙여 `담당자` 로 교체)

### 예시 4 — 괄호 안 코드 리스트 감지 → `코드` 토큰 추가
- 입력: `취소사유(0:단순변심, 1:상품불량, 2:배송누락)`
- 출력: `{"attr": "취소사유(0:단순변심, 1:상품불량, 2:배송누락)", "tokens": ["취소", "사유", "코드"], "confidence": 0.9}`
- 적용 규칙: 원칙 1-C (괄호 내부는 제외, `숫자:한글` 리스트 패턴 감지로
  말미에 `코드` 토큰 추가)

### 예시 5 — 영어 복합 업무 용어 의미 압축
- 입력: `Cart-to-Purchase Rate`
- 출력: `{"attr": "Cart-to-Purchase Rate", "tokens": ["장바구니", "전환율"], "confidence": 0.88}`
- 적용 규칙: 원칙 2-A + 원칙 4-A (`Cart-to-Purchase` 전체를 "전환" 으로
  압축 + `Rate` 를 `율` 로 정규화해 `전환율` 로 병합)

### 예시 6 — 금액 정규화 (금 → 금액)
- 입력: `미수납금`
- 출력: `{"attr": "미수납금", "tokens": ["미수납", "금액"], "confidence": 0.93}`
- 적용 규칙: 원칙 5-A (`미수납` 은 단일 업무 개념) + 원칙 3-B (`금` →
  `금액`)

### 예시 7 — 속성명 전체가 코드 리스트 → 미변환
- 입력: `1:계획, 2:요청`
- 출력: `{"attr": "1:계획, 2:요청", "tokens": [], "confidence": 0.0}`
- 적용 규칙: 원칙 1-E (속성 본체가 이름이 아니라 `숫자:값` 형태의 코드
  정의 자체. 변환 대상이 아니므로 빈 배열 + confidence 0.0)

### 예시 8 — 일반 접미사 `명` 독립 토큰 + 업계 약어 보존
- 입력: `1차BP담당자명`
- 출력: `{"attr": "1차BP담당자명", "tokens": ["1차", "BP", "담당자", "명"], "confidence": 0.92}`
- 적용 규칙: 원칙 4-C (`1차` 숫자+한글 복합) + 원칙 2-D (`BP` 업계 약어
  원본 유지) + 원칙 3-A (`담당` → `담당자`) + 원칙 5-B (접미사 `명` 독립
  토큰)

### 예시 9 — 날짜 정규화 (일 → 일자)
- 입력: `신청시작일`
- 출력: `{"attr": "신청시작일", "tokens": ["신청", "시작", "일자"], "confidence": 0.94}`
- 적용 규칙: 원칙 3-C (날짜 의미의 단일글자 `일` 을 `일자` 로 교체)

### 예시 10 — 괄호 의미 보충 + 복합 토큰 + 연결어 `or`
- 입력: `협력사추천서or1-2차간거래금액증빙서류(첨부파일)`
- 출력: `{"attr": "협력사추천서or1-2차간거래금액증빙서류(첨부파일)", "tokens": ["협력사", "추천서", "1-2차간", "거래", "금액", "증빙", "서류", "첨부", "파일"], "confidence": 0.82}`
- 적용 규칙: 원칙 4-A (연결어 `or` 는 경계로만) + 원칙 4-C (`1-2차간`
  숫자-숫자+한글 복합 유지) + 원칙 5-B (`서류`, `파일` 독립 토큰) +
  원칙 1-D (`(첨부파일)` 은 의미 있는 보충이므로 포함해 분해)

### 예시 11 — 시점 접미사 `전` → `직전` + 단위 괄호 포함
- 입력: `180도소모전력전(kwh)`
- 출력: `{"attr": "180도소모전력전(kwh)", "tokens": ["180도", "직전", "소모", "전력", "KWH"], "confidence": 0.85}`
- 적용 규칙: 원칙 4-C (`180도` 숫자+한글 복합) + 원칙 3-C (시점 접미사
  `전` → `직전`) + 원칙 1-D + 원칙 6-B (`(kwh)` 단위 포함 + 대문자 표준화)

### 예시 12 — ordinal 패턴 (3RD PARTY → 제3자)
- 입력: `3RDPARTY여부`
- 출력: `{"attr": "3RDPARTY여부", "tokens": ["제3자", "여부"], "confidence": 0.88}`
- 적용 규칙: 원칙 2-C (`3RD PARTY` ordinal + 단어 → `제3자`) + 원칙 5-B
  (접미사 `여부` 독립 토큰)

### 예시 13 — 영문 음차 (Aging → 에이징)
- 입력: `Aging구분코드`
- 출력: `{"attr": "Aging구분코드", "tokens": ["에이징", "구분", "코드"], "confidence": 0.8}`
- 적용 규칙: 원칙 2-B (`Aging` 은 업무 번역이 애매 → 음차 `에이징`) +
  원칙 5-B (`구분`, `코드` 독립 토큰)

## 배치 처리 규칙 (시스템 운영용)

형태소분석 커맨드는 아래 규칙으로 속성명을 묶어서 LLM 에 전달합니다. 이
섹션은 배치 크기 튜닝의 근거 자료이며 `morpheme_analyzer.py` 의 상수와
일치해야 합니다.

- **권장 배치 크기: 30** (속성 평균 길이 10~20자 기준)
- **자동 조정 공식**: `max(10, min(50, 1200 // avg_attr_len))`
  - 속성 평균 30자 이상 (긴 복합어 많음) → 20 부근으로 내려감
  - 속성 평균 8자 이하 (짧은 약어) → 50까지 올라감
- **JSON 실패 시 재시도**: 배치 크기를 절반으로 축소해 최대 2단계 재시도
  (30 → 15 → 7). 3회째 실패 시 해당 범위는 파싱 실패로 기록
- **temperature**: 0.0 ~ 0.1 (결정적 응답 필요)
- **timeout**: 배치당 120초
- **병렬도**: 기본 1. `--parallel N` 으로 증가 가능하나 사내 게이트웨이
  rate limit 확인 후 사용

## 배치 크기 trade-off 참고표

| 배치 | 입력 토큰 | 출력 토큰 | 1콜 체감시간 (중형 로컬 LLM) | 리스크 |
|------|-----------|-----------|------------------------------|--------|
| 10   | ~500      | ~400      | 3~6초                        | 호출 오버헤드 지배 |
| 30   | ~800      | ~1,200    | 8~15초                       | 최적 (JSON 안정) |
| 50   | ~1,200    | ~2,000    | 15~25초                      | 긴 출력에서 JSON 깨짐 증가 |
| 100  | ~2,400    | ~4,000    | 30~50초                      | JSON 중간 잘림 빈발 |

## 처리 속도 추정 (2만 건 기준)

|---------|--------|---------------|-------------|-------|

## 품질 체크 (리포트 단계에서 자동 검증)

- `len(tokens) == 0` → 파싱 실패 플래그 (빨강 하이라이트)
- `len(tokens) > 12` → 단어12까지만 저장 + 비고에 "13번째 이후 N개 생략"
  기록 (파랑 하이라이트)
- `confidence < 0.7` 이고 `> 0.0` → 저신뢰도 (노랑 하이라이트, 수동 검토
  큐)

## Few-shot 선정 원칙 (지침 유지관리용)

- **규칙 커버리지**: 하위 규칙 (1-A, 1-B, ..., 6-B) 이 최소 1개 예시로
  커버되어야 한다. 현재 매핑 (13 예시):
  - 1-A (대괄호 제거) → 예시 1
  - 1-C (괄호 내 코드 리스트 → `코드` 추가) → 예시 4
  - 1-D (괄호 내 의미 보충 포함) → 예시 10, 11
  - 1-E (속성명 전체가 코드 리스트 → 미변환) → 예시 7
  - 2-A (업무 용어 번역) → 예시 2, 5
  - 2-B (영문 음차) → 예시 13
  - 2-C (ordinal 패턴) → 예시 12
  - 2-D (업계 약어 보존) → 예시 8 (`BP`)
  - 3-A (사람 역할 정규화) → 예시 3, 8
  - 3-B (금액 정규화) → 예시 6, 10
  - 3-C (시간 / 날짜 정규화) → 예시 9 (`일` → `일자`), 11 (`전` → `직전`),
    5 (`Rate` → `율`)
  - 3-D / 5-B (일반 접미사 분리) → 예시 8, 10, 12, 13
  - 4-A (구분자) → 예시 5, 10
  - 4-B (CamelCase) → 예시 2
  - 4-C (숫자+한글 복합) → 예시 8, 10, 11
  - 5-A (과분해 금지) → 예시 6 (`미수납`), 10 (`협력사`, `증빙`)
  - 6-B (단위 표준화) → 예시 11 (`kwh` → `KWH`)
- **다양성**:
  - 길이: 짧음 (3자, 예시 7) / 중간 (10자) / 긺 (25자+, 예시 10) 모두
  - 언어 조합: 영문전용 (예시 5, 13) / 한글전용 (예시 3, 6, 7, 9) /
    한영혼합 (예시 2, 8, 11, 12) / 괄호/기호 포함 (예시 1, 4, 10, 11)
  - 도메인: 주문 / 배송 / 결제 / 협력사 / 전력 / 에이징 등 이커머스 +
    엔터프라이즈 업무 혼합
- **중복 제거**: 같은 규칙만 반복되는 예시는 1개로 통합
- **순서**: 쉬운 것 → 복잡한 것. 현재 순서 (의도적):
  1-6 = 기본 패턴 (대괄호, 번역, 담당자, 코드리스트, 복합번역, 금액)
  7 = 미변환 엣지 / 8 = 복합 케이스 (숫자+약어+접미사) /
  9 = 날짜 정규화 / 10 = 최고 복잡 (or + 복합 + 괄호) /
  11 = 시점+단위 / 12 = ordinal / 13 = 음차
- **권장 개수**: 10 ~ 14 (엣지 케이스 많아 recency bias 위험 감수).
  현재 13
- **업데이트 주기**: 실제 저신뢰도 사례가 반복 발견되면 해당 케이스를
  예시 14 ~ 로 추가. 추가 시 위 커버리지 매핑표도 함께 갱신
