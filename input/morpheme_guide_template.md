# 형태소 분석 지침 (Morpheme Analysis Guide)

이 파일은 `python main.py morpheme --guide <path>` 커맨드가 LLM 프롬프트에
통째로 주입하는 지침서 **템플릿** 입니다. 내용을 수정하면 분해 품질과 처리
속도가 직접 변합니다. 배치 처리 규칙 섹션도 함께 유지·보수하세요.

사용법:

```bash
cp input/morpheme_guide_template.md input/my_morpheme_guide.md
# my_morpheme_guide.md 를 프로젝트 도메인에 맞게 수정 후
python main.py morpheme --input attrs.txt --guide input/my_morpheme_guide.md
```

`.gitignore` 가 `input/*_template.*` 만 커밋 허용하므로 사용자 버전은 자동으로
gitignore 처리됩니다.

## 역할

당신은 이커머스 / 유통 / 공급망 시스템의 데이터 표준화 전문가입니다.
주어진 속성명을 **도메인 의미 단위** 로 분해해 한글 단어 목록으로
반환합니다. 단순한 글자 단위 토큰화가 아니라, 영문 업무 용어는 한글로
번역하고 의미가 드러나도록 정규화 / 확장까지 수행합니다.

## 기본 원칙

1. **괄호 / 대괄호 제거 (+ 코드 힌트 감지)** — `[]`, `()`, `{}` 안의
   주석·식별자·코드 리스트는 토큰에 포함하지 않는다. 다만 괄호 안이
   `0:단순변심, 1:상품불량` 처럼 **숫자:한글 / 영문:한글 형태의 코드 리스트**
   이면 마지막에 `코드` 토큰을 추가한다.
   - `[abc]주문번호` → `["주문", "번호"]`
   - `취소사유(0:단순변심, 1:상품불량, 2:배송누락)` → `["취소", "사유", "코드"]`
   - `상태(변경)` → `["상태", "변경"]` (단순 보충 설명, `코드` 추가 안 함)

2. **영문 업무 용어 한글 번역** — `Order`, `Checkout`, `Rate`, `Duration`,
   `Cart` 같은 일반 업무 용어는 한글로 치환해 토큰화한다. 단, **고유명사 /
   제품명 / 업계 약어** (`BACKEND`, `FAB`, `PPID`, `LOT`, `MES`, `RFID`,
   `EAI`, `API` 등) 는 원본 그대로 유지하고 토큰으로 보존한다. 영어 복합
   용어 전체가 하나의 업무 개념을 가리키면 의미 단위로 번역·압축한다.
   - `OrderDatetime` → `["주문", "일시"]`
   - `Cart-to-Purchase Rate` → `["장바구니", "전환율"]` (Cart-to-Purchase
     전체를 "전환" 으로 의미 압축)
   - `Checkout Duration` → `["결제", "소요", "시간"]`

3. **의미 확장 / 정규화** — 토큰화 후 아래 세 가지를 일괄 적용한다.
   - **사람 역할 정규화**: 토큰이 `담당`, `책임`, `관리` 처럼 **사람 직무로
     해석되는 단어** 로 끝나면 `자` 를 붙여 `담당자`, `책임자`, `관리자` 로
     교체. 단, 이미 `자` 로 끝나거나 (`사용자`, `관리자`) "관리" 가 시스템
     / 부서를 뜻할 때 (`관리시스템`) 는 변경하지 않는다.
   - **금액 정규화**: 토큰이 `금`, `료`, `비` 로 끝나 **돈을 의미** 하면
     `금액` 으로 교체. 단, `세금`, `보험금` 처럼 이미 굳은 합성어는 유지.
   - **시간 확장**: `Duration`, `Time`, `소요시간` 은 `소요`, `시간` 두
     토큰으로 펼침.
   - 예: `배송담당` → `["배송", "담당자"]` / `미수납금` → `["미수납", "금액"]` /
     `Checkout Duration` → `["결제", "소요", "시간"]`

4. **구분자 + CamelCase 분리** — `-`, `_`, `/`, `.`, 공백은 경계로만
   사용하고 결과에는 포함하지 않는다. CamelCase / PascalCase 는 대문자
   전환점에서 자른다. 연속 대문자는 약어로 묶어 하나의 토큰으로 유지.
   - `order_status_code` → (번역 후) `["주문", "상태", "코드"]`
   - `OrderLineItem` → `["주문", "라인", "항목"]`

5. **한글 단어 보존 (과분해 금지)** — 이미 한글로 된 업무 단어는 어간/어미
   단위로 분해하지 않는다. `설비가동률`, `전환율`, `단순변심` 같은 **단일
   업무 개념** 은 그대로 유지.
   - `설비상태코드` → `["설비", "상태", "코드"]` (O)
   - `설비상태코드` → `["설", "비", "상", "태", "코", "드"]` (X)

6. **업계 약어 / 고유명사 보존** — 약어, 제품명, 시스템명, 외래어 고유명사
   는 번역하지 않고 원본 유지. 판단이 모호하면 confidence 를 0.7 미만으로
   낮춰 수동 검토 큐에 넘긴다.
   - 약어 예: `FAB`, `WIP`, `LOT`, `BOM`, `PPID`, `MES`, `SKU`, `PLU`,
     `EDI`, `API`, `RFID`, `SECS`, `GEM`, `SPA`, `PO`, `SO`.

## 출력 형식

**반드시 유효한 JSON 배열**만 응답하세요. 설명 / 코드펜스 / 부가 텍스트 금지.

```json
[
  {"attr": "[abc]주문번호", "tokens": ["주문", "번호"], "confidence": 0.98},
  {"attr": "OrderDatetime", "tokens": ["주문", "일시"], "confidence": 0.95}
]
```

필드 정의:

- `attr`: 입력 속성명을 그대로 반환 (수정 금지, 원본 괄호/기호 포함)
- `tokens`: 분해 + 번역 + 정규화가 모두 적용된 **최종** 한글 토큰 배열.
  순서는 원본 등장 순서에 맞춘다 (정규화로 토큰이 교체된 경우에도
  위치는 유지)
- `confidence`: 0.0 ~ 1.0. 번역이 모호하거나 / 업계 약어 판단이 흔들리거나 /
  코드 리스트 감지가 불확실하면 0.7 미만으로 낮춰 수동 검토 큐에 넘긴다

## Few-shot 예시 (모델 학습용 — 순서 유지)

아래는 분해 규칙의 적용 예시입니다. LLM 은 이 패턴을 기준으로 동일한 JSON
배열 구조로 응답해야 합니다. 각 예시는 하나 이상의 기본 원칙을 커버합니다.

### 예시 1 — 대괄호 내부 식별자 제거
- 입력: `[abc]주문번호`
- 출력: `{"attr": "[abc]주문번호", "tokens": ["주문", "번호"], "confidence": 0.98}`
- 적용 규칙: 원칙 1 (대괄호 `[abc]` 는 토큰에서 제외, 한글 본문만 분해)

### 예시 2 — 영문 업무 용어 한글 번역
- 입력: `OrderDatetime`
- 출력: `{"attr": "OrderDatetime", "tokens": ["주문", "일시"], "confidence": 0.95}`
- 적용 규칙: 원칙 2 + 원칙 4 (CamelCase 경계 → `Order` / `Datetime`,
  각각 한글 업무 용어로 번역)

### 예시 3 — 사람 역할 정규화 (담당 → 담당자)
- 입력: `배송담당`
- 출력: `{"attr": "배송담당", "tokens": ["배송", "담당자"], "confidence": 0.94}`
- 적용 규칙: 원칙 3 (사람 직무로 해석되는 토큰 `담당` 에 `자` 를 붙여
  `담당자` 로 교체)

### 예시 4 — 괄호 안 코드 리스트 감지 → `코드` 토큰 추가
- 입력: `취소사유(0:단순변심, 1:상품불량, 2:배송누락)`
- 출력: `{"attr": "취소사유(0:단순변심, 1:상품불량, 2:배송누락)", "tokens": ["취소", "사유", "코드"], "confidence": 0.9}`
- 적용 규칙: 원칙 1 (괄호 내부 자체는 토큰 제외) + `숫자:한글` 리스트
  패턴이 감지되므로 말미에 `코드` 토큰 추가. 단어 `단순변심`, `상품불량` 등은
  개별 토큰으로 남기지 않음

### 예시 5 — 영어 복합 업무 용어 의미 압축
- 입력: `Cart-to-Purchase Rate`
- 출력: `{"attr": "Cart-to-Purchase Rate", "tokens": ["장바구니", "전환율"], "confidence": 0.88}`
- 적용 규칙: 원칙 2 + 원칙 4 (하이픈/공백 경계로 `Cart-to-Purchase` 와
  `Rate` 분리. `Cart-to-Purchase` 는 업무적으로 "전환" 1개념이므로
  압축 + `Rate` 는 `율` 로 정규화해 `전환율` 로 병합)

### 예시 6 — 시간 확장 (Duration → 소요 + 시간)
- 입력: `Checkout Duration`
- 출력: `{"attr": "Checkout Duration", "tokens": ["결제", "소요", "시간"], "confidence": 0.92}`
- 적용 규칙: 원칙 2 (Checkout → 결제) + 원칙 3 (Duration → 소요 + 시간
  두 토큰으로 확장)

### 예시 7 — 금액 정규화 (금 → 금액)
- 입력: `미수납금`
- 출력: `{"attr": "미수납금", "tokens": ["미수납", "금액"], "confidence": 0.93}`
- 적용 규칙: 원칙 5 (`미수납` 은 단일 업무 개념, 과분해 금지) + 원칙 3
  (돈을 뜻하는 `금` 을 `금액` 으로 교체)

## 배치 처리 규칙 (시스템 운영용)

형태소분석 커맨드는 아래 규칙으로 속성명을 묶어서 LLM 에 전달합니다. 이
섹션은 배치 크기 튜닝의 근거 자료이며 `morpheme_analyzer.py` 의 상수와
일치해야 합니다.

- **권장 배치 크기: 30** (속성 평균 길이 10~20자 기준)
- **자동 조정 공식**: `max(10, min(50, 1200 // avg_attr_len))`
  - 속성 평균 30자 이상 (긴 복합어 많음) → 20 부근으로 내려감
  - 속성 평균 8자 이하 (짧은 약어) → 50까지 올라감
- **JSON 실패 시 재시도**: 배치 크기를 절반으로 축소해 최대 2단계 재시도
  (30 → 15 → 7). 3회째 실패 시 해당 범위는 파싱 실패로 기록
- **temperature**: 0.0 ~ 0.1 (결정적 응답 필요)
- **timeout**: 배치당 120초
- **병렬도**: 기본 1. `--parallel N` 으로 증가 가능하나 사내 게이트웨이
  rate limit 확인 후 사용

## 배치 크기 trade-off 참고표

| 배치 | 입력 토큰 | 출력 토큰 | 1콜 체감시간 (중형 로컬 LLM) | 리스크 |
|------|-----------|-----------|------------------------------|--------|
| 10   | ~500      | ~400      | 3~6초                        | 호출 오버헤드 지배 |
| 30   | ~800      | ~1,200    | 8~15초                       | 최적 (JSON 안정) |
| 50   | ~1,200    | ~2,000    | 15~25초                      | 긴 출력에서 JSON 깨짐 증가 |
| 100  | ~2,400    | ~4,000    | 30~50초                      | JSON 중간 잘림 빈발 |

## 처리 속도 추정 (2만 건 기준)

| LLM 환경 | 스루풋 | 콜당 (배치30) | 순차 총시간 | 4병렬 |
|---------|--------|---------------|-------------|-------|
| Ollama 중급 CPU | 30 tok/s | ~40초 | 7.4시간 | 1.9시간 |
| vLLM 중급 GPU   | 80 tok/s | ~15초 | 2.8시간 | 42분 |
| 사내 게이트웨이 | 200+ tok/s | ~6초 | 67분 | 17분 |

## 품질 체크 (리포트 단계에서 자동 검증)

- `len(tokens) == 0` → 파싱 실패 플래그 (빨강 하이라이트)
- `len(tokens) > 12` → 단어12까지만 저장 + 비고에 "13번째 이후 N개 생략"
  기록 (파랑 하이라이트)
- `confidence < 0.7` 이고 `> 0.0` → 저신뢰도 (노랑 하이라이트, 수동 검토
  큐)

## Few-shot 선정 원칙 (지침 유지관리용)

- **규칙 커버리지**: 원칙 1 ~ 원칙 6 이 최소 1개 예시로 커버되어야 한다.
  현재 매핑:
  - 원칙 1 (괄호 제거 + 코드 힌트) → 예시 1, 4
  - 원칙 2 (영문 → 한글 번역) → 예시 2, 5, 6
  - 원칙 3 (의미 확장 / 정규화) → 예시 3 (담당자), 6 (소요+시간), 7 (금액)
  - 원칙 4 (구분자 + CamelCase) → 예시 2, 5
  - 원칙 5 (한글 과분해 금지) → 예시 7 (`미수납` 보존)
  - 원칙 6 (약어 / 고유명사 보존) → **예시로 아직 커버 안 됨** → 실제
    운영 데이터에서 `PPID`, `SKU`, `API` 같은 케이스가 저신뢰로 나오면
    예시 8 로 추가 권장
- **다양성**:
  - 길이: 짧음 (3자) / 중간 (10자) / 긺 (20자+) 모두 포함
  - 언어 조합: 영문 전용 / 한글 전용 / 한영 혼합 / 괄호 포함 모두 포함
  - 도메인: 주문 / 배송 / 결제 / 장바구니 / 금액 같은 이커머스 핵심 축
    고르게 분포
- **중복 제거**: 같은 규칙만 반복되는 예시는 1개로 통합
- **순서**: 쉬운 것 → 복잡한 것 (모델이 규칙 난이도를 학습하도록). 현재
  순서: 괄호 제거 → 번역 → 사람 정규화 → 코드 힌트 → 복합 번역 → 시간
  확장 → 금액 정규화
- **권장 개수**: 5 ~ 8 (초과 시 recency bias, 미만 시 규칙 누락). 현재 7
- **업데이트 주기**: 실제 저신뢰도 사례가 반복 발견되면 해당 케이스를
  예시 8 ~ 10 으로 추가. 업계 약어 오분해 케이스는 우선순위 높음
